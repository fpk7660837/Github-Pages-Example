Java threads的简单介绍

进程和线程是并发编程的基本构造模块。

进程自身包含有运行环境。一个进程一般有一批完整私有的基础运行资源，每个进程都有期其自己的内存空间。进程经常被看作异步程序或者应用。然而用户看到的单一应用程序，事实上是一批合作的进程的集合。为了促进进程之间的通信，大多数操作系统支持Inter Process Communication (IPC) resources，比如pipes和sockets.IPC不仅仅是为了同一个系统之间的通信，而是不同系统进程之间的通信。

线程有时被当作轻量级进程。进程和线程都提供了运行环境，但是创建一个线程比创建一个进程所耗费的资源少。

线程在进程内部-每个进程至少含有一个线程。线程共享进程的资源，包括Memory和open files.这样使得线程更加高效，但是却来了隐含的通信问题。多进程执行一个Java平台的一个至关重要的特性。每个应用程序至少含有一个或多个线程。从应用程序的角度来说，通常是由一个main线程开始进而创建更多额外的线程，在接下来的部分会提到，大多数的Java虚拟机的实现都是一个单独的进程。

接下来的部分会假设你已经了解了基本的线程编程方式。

Thread dumps是什么

让我们通过以下的thread dump来了解什么是thread dumps

顾名思义，thread dump就是java虚拟机所有线程的转存文件，它包含了当前虚拟机执行的所有线程的状态。以上thread dump片段展示了2个应用程序线程 [Thread-0 and Thread-1] 和java虚拟机特定的线程，比如“Signal Dispatcher”, “Finalizer”.在这篇文章，我们仅仅关注应用程序线程。

Thread dumps主要用在以下情景当中

  • 得到一个在特定时间点当前应用程序执行的整体印象

  • 揭露一些明显的问题

    • 哪些代码是一直执行的
    • 哪些代码是被悬挂起来的
    • 应用程序的锁与线程同步问题

  • 在生产环境下分析问题，这是一个特别有用的问题

Thread dumps的格式

Thread dumps的格式是随着JSE版本变化的。Sun公司和其他供应商表明了格式在不同的版本是不同的。然而有一个一直不变的是Thread dumps内包含的信息级别。就像以上提到的，thread dumps是java虚拟机状态包含所有应用程序和系统级别线程和监控状态的快照。

在上面的Thread dump片段种，你可以明显看出

  1. Thread dump以”Full thread dump“,后续跟着一系列正在执行的线程
  2. 有2个应用程序线程，分别叫“Thread-1”和“Thread-o”，这些是Java虚拟机为线程提供的默认名字
  3. “Thread-1”在Drop对象调用了Ojbect.wait()**后等待一个通知
  4. 类似的，“Thread-o”在调用了Thread.sleep之后休眠在了一个Condition上面
  5. 在这个特别的例子中，没有任何一个线程处于Runnable状态，因此这个应用程序没有作任何真正的工作

尽管Thread dumps也列出了许多和系统相关的线程，但是我们这里不会对系统线程进行深入的探索

如何获取Thread dumps
